# Lab 08: Wazuh Monitoring Server Setup

## Overview

This lab documents the deployment and configuration of a Wazuh monitoring server for the Project Athenaeum Business Guardian environment.

The Wazuh server was created as the centralized security-monitoring system for the isolated `BusinessGuardianLab` network. It will eventually collect and analyze endpoint telemetry, security events, system logs, file-integrity activity, authentication events, and other information generated by the simulated Windows business workstation.

The server uses two virtual network adapters. One adapter provides temporary outbound connectivity for installation and updates, while the second adapter connects the server to the isolated Business Guardian lab network.

## Objective

Deploy and configure a Wazuh monitoring server that can securely communicate with future monitored endpoints inside the isolated `BusinessGuardianLab` environment while allowing controlled dashboard access from the host computer.

## Static Internal Address Configuration

The internal Wazuh interface was configured with:

```text
Interface: enp0s8
Address: 192.168.70.20/24
Network: 192.168.70.0/24
```

A predictable static address is required because monitored endpoints must know where to send their security data.

The Wazuh server and Windows workstation now occupy the same isolated subnet:

```text
Wazuh server:        192.168.70.20
Windows workstation: 192.168.70.10
```

## Troubleshooting

The lab required troubleshooting several areas, including:

- Identifying the correct Linux interface names
- Distinguishing the NAT adapter from the internal adapter
- Assigning a persistent address to `enp0s8`
- Verifying that the internal network name matched exactly
- Confirming that the virtual cable was connected
- Reaching the Wazuh dashboard from the host browser
- Configuring the correct host and guest ports
- Interpreting the browser certificate warning
- Protecting installation credentials from public evidence

The troubleshooting process reinforced the importance of validating one layer at a time:

1. VirtualBox adapter configuration
2. Linux interface configuration
3. IP addressing
4. Service status
5. Port forwarding
6. Browser access

## Lessons Learned

This lab demonstrated that deploying a monitoring platform requires more than installing software.

The network architecture must allow the server to perform administrative tasks while also maintaining a trusted path to monitored endpoints. Using separate NAT and internal adapters provides a practical way to separate those functions.

The lab also reinforced the value of methodical troubleshooting. Verifying VirtualBox settings, Linux interfaces, IP addresses, services, port-forwarding rules, and browser access individually made it easier to locate and correct configuration problems.

Finally, the lab reinforced that security documentation must protect credentials. Evidence should prove that work was completed without publishing information that could compromise the environment.
